# Auditoría Arquitectónica — URA v1.0.0

**Fecha:** 2026-07-20
**Commit:** `40d5403`
**Propósito:** Due diligence técnica. No propone implementaciones. Solo detecta problemas y documenta hallazgos.

---

## 0. Resumen Ejecutivo

URA v1.0.0 es un sistema funcional con activos valiosos (motor conversacional, 7 proveedores LLM, sistema de memoria, protocolos de plataforma), pero arrastra una deuda arquitectónica severa por la coexistencia de dos codebases paralelos (`core/` y `motor/`), documentación masiva desactualizada, y la ausencia de trazabilidad entre ADRs, implementación y roadmap.

**Número de hallazgos: 4 críticos, 12 altos, 9 medios, 6 bajos.** La mayoría son deuda estructural, no bugs funcionales.

---

## 1. ARQUITECTURA

### 1.1 Capas Innecesarias

**H1 — Dos codebases paralelos: `core/` (204 archivos) vs `motor/` (266 archivos)**

El proyecto tiene dos paquetes raíz con responsabilidades superpuestas:

| Concepto | En `core/` | En `motor/` |
|----------|-----------|-------------|
| Providers LLM | `core/mochila/providers/` (6 providers legacy) | `motor/core/llm/` (7 providers nuevos) |
| Router LLM | `core/model_router.py` (1274 líneas) | `motor/core/llm/router.py` (535 líneas) |
| Config | `core/config_manager.py` (aún activo) | `motor/core/config.py` (UraConfig) |
| Memoria | `core/memoria/` (9 archivos), `core/memory_engine.py` | `motor/memory/` (F26), `motor/intelligence/memory/` (F12) |
| Agentes | — | `motor/agents/` (F27) + `motor/intelligence/agents/` (F12) |
| Executor | (eliminado) | `motor/core/executor.py` |
| Circuit breaker | `core/mochila/circuit_breaker.py` | `motor/core/llm/circuit_breaker.py`, `motor/diagnostico/circuit_breaker.py` |

**Evidencia:** 9 archivos comparten nombre exacto entre `core/` y `motor/`: `gemini.py`, `ollama.py`, `openrouter.py`, `router.py`, `bridge.py`, `chunking.py`, `circuit_breaker.py`, `engine.py`, `base.py`.

**Conclusión:** `core/` es un legado que debió eliminarse o congelarse al migrar a `motor/`. En lugar de eso, siguió recibiendo imports desde `motor/` y viceversa, creando un acoplamiento circular.

**H2 — Dos sistemas de agentes paralelos**

```
motor/agents/           → F27 (nuevo: gate, scheduler, runner, planner, state)
motor/intelligence/agents/ → F12 (viejo: consensus, researcher, supervisor, validator, executor, planner)
```

Ambos tienen `base.py`, `planner.py`, `executor.py`/`runner.py`. El roadmap no documenta cuál es el canónico ni si el viejo debe deprecarse.

**H3 — Dos sistemas de memoria paralelos**

```
motor/memory/               → F26 (journal, snapshot, timeline, crypto)
motor/intelligence/memory/  → F12 (episodic, semantic, compression, forgetting, retrieval)
```

Misma situación: el roadmap no aclara si coexisten o si uno debe reemplazar al otro.

### 1.2 Sobreingeniería

**H4 — `motor/platform/` (16 archivos) con 1 solo consumidor externo**

- Archivos: `tracing.py` (894 líneas), `transport.py`, `validator.py`, `negotiator.py`, `serializer.py`, `middleware.py`, `delivery.py`, `audit.py`, `metrics.py`, `registry.py`, `models.py`, `errors.py`, etc.
- Consumidores externos: `scripts/soak_test.py` + 4 tests.
- Esto es 1 script y 4 tests para 16 archivos de infraestructura de protocolo. Indica que la capa de protocolos (F28) se construyó completa antes de tener consumidores reales.

**H5 — `motor/intelligence/agents/` (11 archivos): agentes que nadie usa**

- Define `ConsensusEngine`, `ParallelExecutor`, `ReflectionExecutor`, `SupervisorAgent`, `ResearcherAgent`, `ValidatorAgent`, `PlannerAgent`.
- Buscando consumidores: estos se testean en `tests/` pero no hay evidencia de que los agentes F12 se usen en producción. El runtime real probablemente usa `motor/assistant/` que no depende de `motor/intelligence/agents/`.

**H6 — 37 ADRs para 29 fases (1.3 ADRs por fase)**

Muchas fases pequeñas (F5-F8, F14-F17) generaron ADRs y closeouts para cambios mínimos. La fase 28 tiene 10 ADRs para una sola capa de protocolo que apenas tiene consumidores. Esto sugiere sobre-documentación como sustituto de trazabilidad real.

### 1.3 Responsabilidades Mezcladas

**H7 — `core/config_manager.py` es una fuente de verdad no declarada**

Importado por:
- `motor/core/llm/ollama.py` (el único provider que no migró a `UraConfig`)
- `motor/core/llm/__init__.py`
- `motor/core/config.py`
- `core/memory_engine.py`
- `core/auto_reindex.py`
- `core/debate/debate_engine.py`
- `core/logs/guardian_logger.py`

`UraConfig` en `motor/core/config.py` debía ser la fuente de verdad única (F17), pero `config_manager.py` sigue activo y es usado por ambos lados. `UraConfig` incluso importa `CONFIG` desde `config_manager.py`.

**H8 — `core/memory_engine.py` toca 3 subsistemas distintos**

Importa `motor.core.config`, `motor.core.llm`, `motor.core.qdrant_client` — mezcla config, LLM y vector store en un solo archivo (636 líneas). Debería ser 3 responsabilidades separadas.

### 1.4 Violaciones SOLID

**H9 — Principio de Inversión de Dependencias violado**

```
motor/ → core/config_manager   (la capa "nueva" depende de la "vieja")
core/  → motor/core.*          (la capa "vieja" depende de la "nueva")
```

Esto es un ciclo bidireccional: `/motor/core/llm/ollama.py` → `core.config_manager` Y `core/memory_engine.py` → `motor.core.llm.generate`. Ninguna capa es independiente.

**H10 — Single Responsibility: `motor/platform/tracing.py` (894 líneas)**

894 líneas para tracing. Comparación: el módulo de tracing de OpenTelemetry tiene ~20 clases separadas. Este archivo mezcla span management, export, sampling, propagación y sanitización en una sola clase.

**H11 — Open/Closed: providers duplicados**

Para añadir un nuevo provider LLM, necesitas:
1. Crear archivo en `motor/core/llm/`
2. Crear (o no) archivo en `core/mochila/providers/`
3. Registrar en `motor/core/llm/__init__.py`
4. Posiblemente registrar en `core/mochila/router.py`
5. Añadir tests en `motor/tests/`

Son 5 puntos de contacto para lo que debería ser 1 (un registry).

### 1.5 Acoplamiento Excesivo

**H12 — `motor/platform/` tiene acoplamiento interno extremo**

De los 16 archivos, 14 se importan entre sí. `validator.py` importa 4 sub-módulos de platform. `delivery.py` importa 3. `transport.py` importa 3. El acoplamiento es tan alto que `motor/platform/` es esencialmente un monolito partido en 16 archivos, no 16 módulos independientes.

**H13 — Ciclo directo detectado**

```
motor/cli/cmd_ura.py → core.memory_engine → motor.core.config, motor.core.llm, motor.core.qdrant_client
```

Esto es un ciclo en producción: el CLI → engine → config/llm/qdrant → vuelta a motor. Rompe toda posibilidad de testing aislado del CLI.

## 2. CÓDIGO

### 2.1 Código Muerto

**H14 — `motor/intelligence/agents/` (11 archivos) sin consumidores conocidos en producción**

Los agentes F12 (`consensus.py`, `supervisor.py`, `researcher.py`, etc.) existen pero el asistente conversacional (`motor/assistant/`) no depende de ellos. Si no hay consumidor, es código muerto mantenido.

**H15 — `core/mochila/providers/deepseek.py` y `groq.py`**

DeepSeek y Groq solo existen en la capa legacy (`core/mochila/providers/`). No hay versión en `motor/core/llm/`. Si `core/mochila/` es legado por deprecar, estos providers se perderán.

**H16 — 3 versiones de `AUDIT_FASE7_DESIGN`**

`AUDIT_FASE7_DESIGN.md`, `AUDIT_FASE7_DESIGN_V2.md`, `AUDIT_FASE7_DESIGN_V3.md` — coexisten. Solo la V3 debería existir.

**H17 — `core/modules/` directorio vacío**

Contiene subdirectorios pero 0 archivos .py. Carga de mantenimiento sin propósito.

### 2.2 Código Duplicado

**H18 — `core/mochila/providers/*.py` vs `motor/core/llm/*.py`**

Comparación directa:

| Provider | core/mochila/providers/ | motor/core/llm/ |
|----------|------------------------|-----------------|
| Ollama | `ollama.py` | `ollama.py` |
| Gemini | `gemini.py` | `gemini.py` |
| OpenRouter | `openrouter.py` | `openrouter.py` |
| Base class | `base.py` | `base.py` |
| Router | `router.py` | `router.py` |
| Circuit breaker | `circuit_breaker.py` | `circuit_breaker.py` |

Son 6 conceptos duplicados en 2 ubicaciones. Cada par probablemente comparte ~60-80% de lógica.

**H19 — 10 archivos llamados `base.py`, 7 llamados `models.py`, 6 llamados `registry.py`**

En un proyecto de este tamaño, tener nombres genéricos dificulta la navegación. Un nuevo desarrollador no sabe qué `base.py` abrir.

**H20 — 773 `except Exception:` en toda la codebase**

Esto es sistémico. Aunque muchos son degradación controlada, el número indica que no hay una política clara de manejo de errores. Cada módulo decide cómo atrapar excepciones.

### 2.3 APIs Redundantes

**H21 — `core/model_router.py` (1274 líneas) y `motor/core/llm/router.py` (535 líneas)**

Ambos hacen routing de modelos LLM. `core/model_router.py` es un servidor HTTP completo con dashboard. `motor/core/llm/router.py` es una clase Library con circuit breaker. La lógica de selección de modelo (qué modelo para qué tarea) está duplicada.

**H22 — `motor/assistant/evaluation.py` y `motor/core/evaluation/`**

Ambos hacen evaluación pero a niveles distintos (asistente vs RAG). La nomenclatura no lo aclara.

### 2.4 Imports Innecesarios

**H23 — `pyproject.toml` versión 0.13.0 vs realidad**

`pyproject.toml:11` declara `version = "0.13.0"`. Los tags reales llegan a `v0.29.0-fase29`. La versión de `motor/assistant/main.py` es `1.0.0`. Hay **tres versiones distintas** en el mismo repositorio.

Además: `rank-bm25` y `blake3` están en `pyproject.toml` pero no en `requirements/base.txt`. `typer` y `coloredlogs` están en `requirements/base.txt` pero no en `pyproject.toml`. Dos fuentes de verdad desincronizadas.

### 2.5 Dependencias Circulares

**H24 — Ciclo bidireccional `core/` ↔ `motor/`**

Ver H9 y H13. 13 archivos en `core/` importan de `motor/`. 1 archivo en `motor/` importa de `core/`. Aunque la dirección dominante es `core/` → `motor/`, la existencia de la ruta inversa (`motor/` → `core/config_manager`) crea un ciclo que impide separar los paquetes.

## 6. RENDIMIENTO

### 6.1 Complejidad Innecesaria

**H28 — `motor/platform/serializer.py` define un serializador JSON canónico**

Para un sistema que podría usar `orjson` o `ujson` directamente, define 268 líneas de serialización con validación de esquemas. ¿El beneficio real vs `orjson.dumps()`? No está documentado.

**H29 — `motor/platform/negotiator.py` (114 líneas)**

Negociación de versión de protocolo para 0 consumidores externos. Si `motor/platform/` apenas se usa, este código se ejecuta 0 veces en producción.

### 6.2 Objetos Creados sin Necesidad

**H30 — 25 `__init__.py` vacíos**

Cada uno crea un objeto package en memoria para cero funcionalidad.

### 6.3 Cachés Inútiles

**H31 — Caché de prompts en `core/model_router.py`**

`PromptCache` con TTL de 2h y 5min. Si el router es un proxy hacia Ollama local (latencia <100ms), ¿qué problema resuelve una caché de prompts? El LLM ya tiene KV cache.

**H32 — `motor/intelligence/reranking/ce.py` y `reranker.py`**

Cada CrossEncoderReranker tiene `_load_docs()` que carga documentos golden en memoria. Estos documentos se usan para reranking. Pero el reranking solo ocurre durante evaluación, no en producción. La carga de documentos ocurre en `__init__`, incluso si el reranker nunca se usa.

## 10. RESULTADO

### 10.1 Fortalezas

1. **`motor/core/llm/`** está bien diseñado: 7 providers, registry, circuit breaker, retry, observabilidad. Es la parte más madura del proyecto.
2. **`motor/assistant/`** es un producto funcional: FastAPI, 31 archivos, evaluación, contexto, preferencias. El health endpoint responde.
3. **Audit reproducible**: `scripts/audit_reproducible.sh` pasa 12/12 checks desde un clone limpio. Esto es más de lo que la mayoría de proyectos puede decir.
4. **Test count**: 122 tests pasan sin fallos. No es cobertura alta, pero los que existen son verdes.
5. **Ruff 0 errores**: El proyecto está limpio de linting. Esto es raro en proyectos de ~600K LOC.
6. **Documentación exhaustiva**: Aunque es excesiva, hay documentación para casi todo. Un nuevo desarrollador puede entender el roadmap (aunque tenga contradicciones).

### 10.2 Debilidades

1. **Arquitectura partida en dos**: `core/` vs `motor/` es la deuda más costosa. Cada nuevo desarrollador pierde tiempo decidiendo dónde poner el código.
2. **Sin CI/CD real**: Los workflows de GitHub Actions existen pero nunca se ejecutaron. No hay validación automática en pushes.
3. **Documentación sin mantenimiento**: ~195 .md, muchos obsoletos. La documentación creció más que el código.
4. **Sin política de errores**: 773 `except Exception:` es un olor sistémico. No hay jerarquía de excepciones propia.
5. **Sin trazabilidad ADR→código**: 37 ADRs pero no se puede verificar cuáles están implementados.
6. **pyproject.toml desactualizado**: 0.13.0 cuando el proyecto está en 0.29.0+/1.0.0. Esto rompe cualquier integración con PyPI, dependabot, etc.

### 10.3 Deuda Técnica (Resumen)

| Tipo | Ítems | Costo Estimado |
|------|-------|---------------|
| Código duplicado (core/ vs motor/) | ~20 archivos | 40-60h de consolidación |
| Documentación obsoleta | ~60 archivos | 15-20h de archivo/limpieza |
| ADRs sin finalizar | 8 archivos | 4-8h de revisión y aprobación |
| Tests faltantes (core/ + knowledge/) | 180 archivos sin test en-directorio | 100-200h (no urgente) |
| Versiones inconsistentes | 3 fuentes | 1h de corrección |
| Artefactos en raíz | ~30 archivos | 2h de limpieza |

### 10.4 Riesgos Futuros

1. **Si no se unifica `core/` → `motor/`**, la deuda de mantenimiento crece O(n²): cada nuevo provider requiere implementación en ambos lados.
2. **Si no se ejecuta CI/CD**, no hay detección temprana de regresiones. Los 122 tests solo protegen lo que cubren.
3. **Si la documentación sigue creciendo**, alcanzará un punto donde mantenerla cuesta más que mantener el código.
4. **Si no se resuelven los ADRs en Draft**, la arquitectura de F27-F29 no tiene validez contractual. Cualquier cambio futuro puede ignorar las decisiones documentadas.
5. **El token GitHub en `.git/config`** es un riesgo de seguridad inmediato. Cualquier script que lea el remote URL expone el token.

### 10.5 Recomendaciones para v1.1.0

> No propongo implementar nada. Solo establezco condiciones que, si se cumplen, harían de v1.1.0 una versión más mantenible que v1.0.0.

1. **Unificar `core/` en `motor/`** como prerrequisito para cualquier feature nueva. Si un archivo de `core/` tiene consumidores, migrarlo a `motor/`. Si no, eliminarlo o archivarlo.

2. **Unificar versionado**: `pyproject.toml`, `motor/assistant/main.py` y el tag de git deben coincidir. Decidir si se usa semver (1.x) o el esquema 0.x histórico.

3. **Finalizar ADRs pendientes**: Los 8 ADRs en Draft (F27, F28.1, F29) deben pasar a Approved o las fases deben abrirse de nuevo.

4. **Reducir documentación**: Archivar closeouts de fases cerradas. Consolidar documentos duplicados. Mantener solo: ADRs vigentes, 1 documento de arquitectura, 1 de API, 1 de deploy, el closeout de la versión actual.

5. **Establecer política de imports**: `motor/` no importa de `core/`. Fin. `core/` puede importar de `motor/` durante la migración, pero no al revés.

6. **Ejecutar CI/CD**: Los workflows de GitHub Actions existen pero nunca se ejecutaron. Activar `ci.yml` en cada push a main.

7. **Proteger la rama `main`**: Requerir audit reproducible + CI verde antes de cualquier merge.

8. **No añadir nuevas capas sin consumidor real**: La lección de `motor/platform/` (16 archivos, 1 script que los usa) es que construir infraestructura anticipada genera código muerto.

9. **Resolver el token GitHub en `.git/config`**: Rotar el token y usar `git remote set-url` sin credenciales en la URL.
