## 1. Reanudación tras interrupción

**Problema:** Si el pipeline se interrumpe (apagado, crash, timeout), la siguiente ejecución empieza desde la fase 1, repitiendo trabajo ya completado.

**Solución propuesta:**
- Cada fase registra su estado en `.nervioso/pipeline_state.json`:
  ```json
  {"fase": "refactor", "estado": "completado", "timestamp": "..."}
  ```
- Al iniciar, PipelineEngine lee el estado y salta las fases ya completadas.
- `--force` para ignorar el estado y ejecutar completo.

**Impacto:** Bajo — añadir `checkpoint` + `resume` en engine.py (~30 líneas).

---

## 2. Transacciones del pipeline

**Problema:** Cada plugin ejecuta, pero no hay un ciclo uniforme prepare/execute/validate/commit/rollback. Si un plugin falla, no se revierte su trabajo.

**Solución propuesta:**
- Contrato para plugins:
  ```python
  PLUGIN = {
      "name": "...",
      "phase": "...",
      "transactional": True,  # requiere prepare/commit/rollback
  }
  ```
- `PipelineEngine.run_phase()` ejecuta:
  1. `prepare()` — guardar estado previo (snapshot)
  2. `execute()` — ejecutar plugin
  3. `validate()` — verificar resultado
  4. `commit()` — confirmar cambios
  5. `rollback()` — restaurar si validate falla

**Impacto:** Medio — requiere modificar `plugin_registry.py` y el contrato de plugins. Los plugins existentes seguirían funcionando con `transactional: False`.

---

## 3. Presupuesto de cambios

**Problema:** No hay límite de cambios por ejecución. Un plugin podría modificar 500 archivos sin control.

**Solución propuesta:**
- Añadir a `PromotionPolicy`:
  ```python
  BUDGET = {
      "max_files": 50,
      "max_lines": 5000,
      "max_modules": 10,
  }
  ```
- Después de refactor, contar cambios con `git diff --stat`.
- Si se supera el presupuesto, el pipeline no promociona automáticamente (requiere revisión manual).

**Impacto:** Bajo — añadir `_check_budget()` en engine.py y actualizar PromotionPolicy (~20 líneas).

---

## 4. Tendencias históricas

**Problema:** MetricsCollector guarda datos pero no los analiza. No hay detección de degradaciones.

**Solución propuesta:**
- Añadir `TrendAnalyzer` que lea `historial_ejecuciones.jsonl` y calcule:
  - Duración media por plugin
  - Tasa de fallos
  - Frecuencia de rollback
  - Número de cambios por ejecución
- Incluir en el informe final si se detecta degradación:
  ```json
  {"alerta": "Plugin X es 3x más lento que la media histórica"}
  ```

**Impacto:** Bajo — añadir `scripts/pro/tuneladora/trends.py` (~60 líneas).

---

## 5. Clasificación de fallos

**Problema:** Todos los errores se tratan igual. Timeout de red se maneja como bug de código.

**Solución propuesta:**
- Añadir clasificación al `run_script()`:
  | Tipo | Ejemplo | Acción |
  |------|---------|--------|
  | Transitorio | Timeout de red, 503 | Reintentar (hasta 3 veces) |
  | Configuración | Archivo no encontrado, permisos | Detener y notificar |
  | Bug del código | SyntaxError, AssertionError | Rollback |
  | Infraestructura | Ollama caído, disco lleno | Degradar y continuar |

- PipelineEngine.apply_error_policy(returncode, stderr) → decide acción.

**Impacto:** Medio — requiere añadir `_classify_error()` + `_retry()` en engine.py (~40 líneas).
