# v3.0 — Autonomía

**Estado:** Planificado  
**Depende de:** v2.3 (infraestructura congelada por ADR-030)  
**Principio:** Toda nueva capacidad como plugin o componente desacoplado.

---

## Objetivo

URA recibe un objetivo de alto nivel, lo planifica, lo ejecuta, evalúa el
resultado y decide el siguiente paso con supervisión mínima.

Todo estado persiste en el ExecutionLedger.  
Toda acción tiene límites de presupuesto (tiempo, recursos, coste, cambios).  
Todas las decisiones son auditables.

---

## Arquitectura

```
Objetivo de alto nivel
        │
        ▼
┌──────────────────┐
│   Goal Manager   │  ── crear, priorizar, suspender, cancelar
│   (Fase 1)       │  ── dependencias entre objetivos
└────────┬─────────┘
         │ objetivos activos
         ▼
┌──────────────────┐
│    Planner       │  ── descomponer en tareas
│   (Fase 2)       │  ── ordenar, estimar, seleccionar estrategia
└────────┬─────────┘  ── replanificar si falla
         │ plan de tareas
         ▼
┌──────────────────┐
│  Task Executor   │  ── ejecutar tareas
│   (Fase 3)       │  ── seleccionar herramientas
└────────┬─────────┘  ── reintentos, registro
         │ resultados
         ▼
┌──────────────────┐
│   Evaluator      │  ── medir cumplimiento
│   (Fase 4)       │  ── decidir: finalizar / continuar / corregir
└────────┬─────────┘
         │ lecciones
         ▼
┌──────────────────┐
│  Learning Plugin │  ── analizar ExecutionLedger
│   (Fase 5)       │  ── patrones de éxito/fallo
└──────────────────┘  ── propuestas de mejora

         ┌──────────────────────┐
         │  ExecutionLedger     │  ← todo persiste aquí
         │  (infraestructura    │
         │   v2.3 congelada)    │
         └──────────────────────┘
```

---

## Fase 1 — Goal Manager

Responsabilidad:
- Crear objetivos con descripción, prioridad y presupuesto
- Priorizar activos, suspender, reanudar, cancelar
- Gestionar dependencias entre objetivos
- Persistir en ExecutionLedger

Formato de objetivo:
```json
{
  "id": "obj-001",
  "descripcion": "Reducir deuda técnica en core/",
  "prioridad": 0.8,
  "estado": "activo",
  "dependencias": [],
  "presupuesto": {"tiempo_max_s": 3600, "cambios_max": 50},
  "created_at": "...",
  "updated_at": "..."
}
```

## Fase 2 — Planner

Responsabilidad:
- Descomponer objetivos en tareas atómicas
- Ordenar tareas según dependencias
- Estimar coste y tiempo por tarea
- Seleccionar estrategia (herramientas, plugins, scripts)
- Replanificar cuando falle una tarea (ruta alternativa)

Formato de tarea:
```json
{
  "id": "task-001",
  "objetivo_id": "obj-001",
  "descripcion": "Ejecutar ruff --fix en core/",
  "herramienta": "ruff_fix",
  "estado": "pendiente",
  "estimacion_s": 120,
  "reintentos": 0,
  "max_reintentos": 3
}
```

## Fase 3 — Task Executor

Responsabilidad:
- Ejecutar tareas usando herramientas disponibles
- Seleccionar herramienta según la tarea (run_script, run_ruff, pipeline_refactor)
- Gestionar reintentos (transitorio → reintentar, bug → escalar)
- Registrar resultados en ExecutionLedger
- Respetar presupuestos (si se agota el tiempo, abortar)

Política de reintentos:
| Error | Acción |
|-------|--------|
| Timeout de red | Reintentar hasta 3 veces con backoff |
| Recurso no disponible | Esperar y reintentar |
| Error de validación | Escalar al Evaluator |
| Bug de código | Registrar y abortar tarea |

## Fase 4 — Evaluator

Responsabilidad:
- Medir si el objetivo se ha cumplido (criterios: ruff OK, tests OK, cambios < presupuesto)
- Calcular métricas de calidad del resultado
- Decidir: finalizar ✅ / continuar 🔄 / corregir 🔧 / solicitar intervención 🙋

## Fase 5 — Learning Plugin

Responsabilidad:
- Analizar el historial del ExecutionLedger
- Identificar patrones de éxito y fallo por tipo de tarea
- Proponer mejoras: ajustar timeouts, cambiar herramientas, evitar patrones de fallo
- Actualizar conocimiento procedimental (recetas para tareas recurrentes)

---

## Presupuestos globales

Cada ejecución de autonomía tiene límites:

| Presupuesto | Límite por defecto | Acción al superar |
|-------------|-------------------|-------------------|
| Tiempo total | 3600s | Replanificar o abortar |
| Llamadas a modelo | 50 | Degradar a respuestas rápidas |
| Cambios (archivos) | 50 | Requerir revisión manual |
| Cambios (líneas) | 5000 | Idem |
| Memoria (RAM) | 32GB | Degradar workers |

---

## Lo que NO incluye v3.0

- Descubrimiento automático de conocimiento nuevo
- Colaboración entre múltiples agentes especializados
- Aprendizaje profundo sobre largos periodos
- Modificación autónoma de la infraestructura

Esas capacidades pertenecen a v3.1, v3.2 y v3.3 respectivamente.

---

## Criterios de aceptación

URA debe ser capaz de:

- [ ] Recibir un objetivo complejo (ej: "refactorizar funciones grandes en core/")
- [ ] Dividirlo automáticamente en subtareas
- [ ] Ejecutarlas en el orden adecuado
- [ ] Recuperarse de errores recuperables (timeout, 503)
- [ ] Replanificar cuando cambien las condiciones (objetivo cancelado, recurso no disponible)
- [ ] Registrar todas las decisiones en el ExecutionLedger
- [ ] Finalizar con un informe reproducible

---

## Métricas de éxito (capacidades observables)

No se mide por número de componentes, sino por capacidades demostrables:

| Capacidad | Criterio de éxito |
|-----------|------------------|
| Goal Manager | Mantiene y prioriza múltiples objetivos correctamente |
| Planner | Genera planes válidos y replantea cuando cambian las condiciones |
| Task Executor | Ejecuta tareas con recuperación ante fallos y dentro del presupuesto |
| Evaluator | Decide correctamente entre finalizar, continuar, corregir o escalar |
| Learning Plugin | Produce mejoras verificables usando el historial del ExecutionLedger |

## Estrategia de implementación

No desarrollar cada componente por separado e integrar al final.
En su lugar, construir un **vertical slice** completo desde el día 1:

1. Goal Manager mínimo
2. Planner mínimo
3. Executor mínimo
4. Evaluator mínimo
5. Learning mínimo

Todo integrado desde el primer commit. Después se mejora cada pieza
de forma incremental.

Esto evita el riesgo de tener 5 componentes avanzados que no encajan.

## Primer objetivo integrado (vertical slice)

> Auditar el repositorio URA y generar un informe de deuda técnica priorizado.

Este objetivo obliga a ejercitar todas las capacidades:

| Fase | Acción |
|------|--------|
| Goal Manager | Registrar objetivo "auditar repositorio" |
| Planner | Descomponer en tareas: ruff, inspección, informes |
| Executor | Ejecutar herramientas (ruff, inspectores, revisor) |
| Evaluator | Verificar si el informe se generó correctamente |
| Learning | Registrar patrón para futuras auditorías |
| Ledger | Todo queda registrado: decisiones, resultados, informe |

Si este flujo funciona de extremo a extremo, se valida la arquitectura
funcional completa desde el primer incremento.

## Dependencia con infraestructura

v3.0 usa exclusivamente la API pública de v2.3:

| Componente v2.3 | Uso en v3.0 |
|----------------|-------------|
| `PipelineEngine.run_script()` | Ejecutar tareas como scripts |
| `PipelineEngine.run_ruff()` | Ejecutar tareas de calidad |
| `ExecutionLedger` | Persistir objetivos, tareas, decisiones, resultados |
| `pipeline_refactor.py` | Herramienta para tareas de refactorización |
| `PromotionPolicy` | Evaluar si un objetivo se cumplió |
| `SnapshotService` | Checkpoint antes de cada modificación |
