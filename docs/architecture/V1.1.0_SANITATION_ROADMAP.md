## 0. Filtrado de Hallazgos: Reales vs Falsos Positivos

De los 31 hallazgos de la auditoría, los siguientes son **falsos positivos o ruido aceptable**:

| Hallazgo | Motivo | Acción |
|----------|--------|--------|
| H19 — 10 archivos `base.py` | Convención estándar en Python: cada paquete tiene su `base.py`. No es un problema real. | Ignorar. |
| H26 — 63 `__init__.py` vacíos | Estándar de Python para marcar paquetes. Solo los vacíos en `motor/` raíz podrían re-exportar, pero no es obligatorio. | Ignorar. |
| H30 — 25 `__init__.py` "crean objetos en memoria" | Un `__init__.py` vacío no crea objetos medibles. Falso positivo. | Ignorar. |
| H31 — Prompt cache en model_router | La caché evita llamadas HTTP a Ollama aunque sea local. Tiene sentido para modelos grandes (42GB). | Falso positivo. Mantener. |
| H22 — evaluation.py vs motor/core/evaluation/ | Son niveles distintos (asistente vs RAG). La nomenclatura es confusa pero no es duplicación real. | Bajo. Renombrar si sobra tiempo. |
| H6 — 37 ADRs para 29 fases | No es sobreingeniería. Es documentación de decisiones. Prefiero esto a la falta de ADRs. | Falso positivo. Ignorar. |
| M2 — 54 `# type: ignore` | En un proyecto de 596K LOC con dependencias sin stubs (transformers, torch), 54 supresiones es aceptable. | Ignorar. |
| B6 — 8 `# nosec B615` en HF models | Decisión consciente: modelos conocidos de HuggingFace, no hay revision específica que pinner. | Ignorar. |
| H32 — reranker carga documentos en __init__ | Patrón lazy-load documentado. Si no se usa, la carga no ocurre hasta el primer rerank. | Falso positivo. |

**Falsos positivos confirmados: 9 hallazgos.** Los elimino del roadmap.

**Hallazgos reales: 22.** Todos los demás son problemas legítimos que requieren acción.

## 1. Reordenación por Impacto Arquitectónico

Reordeno los 22 hallazgos reales según su impacto en la arquitectura, no por severidad documental:

### 1.1 Bloqueante: Dualidad core/motor (raíz de casi todos los demás problemas)

```
I1  core/ y motor/ son dos codebases paralelos con responsabilidades superpuestas
    → 470 archivos, 9 con mismo nombre, providers duplicados
    → Causa raíz: nunca se completó la migración de core → motor
    → Impacto hoy: cada nuevo provider necesita implementarse en 2 lugares
    → Riesgo futuro: la deuda crece O(n²) con cada feature nueva
    → Dependencias: ninguna (es el problema raíz)
    
I2  Ciclo bidireccional core/ ↔ motor/ (13 archivos core→motor + 1 motor→core)
    → Causa raíz: core/config_manager.py no migró a motor/core/config.py
    → Impacto hoy: no se puede testear core/ sin motor/ ni viceversa
    → Riesgo: impide separar los paquetes en el futuro
    → Dependencias: depende de I1 (al unificar core→motor, el ciclo desaparece)

I3  core/config_manager.py es fuente de verdad no declarada (7 consumidores)
    → Causa raíz: F17 prometió UraConfig como fuente única pero no migró todos los consumidores
    → Impacto hoy: UraConfig es una fachada que envuelve config_manager.py
    → Riesgo: cualquier cambio de config requiere tocar 2 sistemas
    → Dependencias: se resuelve en I1 (unificar en motor/core/config.py)

I4  core/mochila/providers/ vs motor/core/llm/ (6 providers duplicados)
    → Causa raíz: migración a medio completar
    → Impacto hoy: 6 implementaciones paralelas que hay que mantener sincronizadas
    → Riesgo: divergencia silenciosa (un provider se actualiza en un lado pero no en el otro)
    → Dependencias: I1 (al unificar, se decide cuál es el canónico)
```

### 1.2 Estructural: Sistemas duplicados

```
I5  motor/agents/ (F27) vs motor/intelligence/agents/ (F12) — agentes paralelos
    → Causa raíz: F12 creó agentes, F27 los rediseñó, pero F12 no se eliminó
    → Impacto hoy: 19 archivos, dos planners, dos executors
    → Riesgo: un desarrollador nuevo implementa en el sistema equivocado
    → Dependencias: I1 (después de unificar core/motor, decidir cuál es el canónico)

I6  motor/memory/ (F26) vs motor/intelligence/memory/ (F12) — memorias paralelas
    → Causa raíz: misma historia que I5
    → Impacto hoy: 15 archivos, dos sistemas de memoria que no se comunican
    → Riesgo: datos de memoria en dos silos incompatibles
    → Dependencias: I1

I7  core/model_router.py (HTTP server, 1274 líneas) vs motor/core/llm/router.py (library, 535 líneas)
    → Causa raíz: el router HTTP (servicio systemd) y el router library se construyeron por separado
    → Impacto hoy: lógica de selección de modelo duplicada
    → Riesgo: el router HTTP y el library pueden seleccionar modelos diferentes
    → Dependencias: I1 (unificar en motor/)
```

### 1.3 Gobernanza: ADRs y trazabilidad

```
I8  8 ADRs en Draft para fases marcadas cerradas (F27, F28.1, F29)
    → Causa raíz: se cerraron las fases sin aprobar los ADRs
    → Impacto hoy: las decisiones arquitectónicas de F27-F29 no tienen validez contractual
    → Riesgo: cualquier cambio futuro puede ignorar ADRs que "nunca se aprobaron"
    → Dependencias: ninguna (proceso documental independiente)

I9  FASE16_PROPOSAL.md referenciado pero no existe
    → Causa raíz: error de documentación al escribir AGENTS.md
    → Impacto hoy: link roto en el documento principal del proyecto
    → Dependencias: ninguna

I10 Fases 18-24 sin entrada en roadmap
    → Causa raíz: el roadmap de AGENTS.md saltó de F17.5 a F25
    → Impacto hoy: un desarrollador no sabe qué pasó entre F17.5 y F25
    → Dependencias: I8 (misma family de problemas de gobernanza)

I11 Sin trazabilidad ADR → implementación
    → Causa raíz: no hay mecanismo para verificar que un ADR se implementó
    → Impacto hoy: ADR-028-04 requiere checksum SHA-256, pero F28_B2_CODE_AUDIT.md dice que "checksum nunca se verifica"
    → Dependencias: ninguna (pero es estructural, no se resuelve rápido)

I12 Closeouts faltantes: F14, F16, F17, F17.5, F24
    → Causa raíz: se cerraron las fases sin generar el documento de cierre
    → Impacto hoy: no hay registro de qué se hizo en esas fases
    → Dependencias: ninguna
```

### 1.4 Sobreingeniería con Consumidores Reales

```
I13 motor/platform/ (16 archivos, ~3000 LOC) — 1 consumidor externo
    → Causa raíz: F28 construyó una capa de protocolo completa antes de tener consumidores

    → Riesgo: si nunca se adopta, es el mayor caso de sobreingeniería del proyecto

I14 motor/intelligence/agents/ (F12) — 11 archivos sin consumidores conocidos en producción
    → Causa raíz: F12 creó agentes que F27 reemplazó, pero no se eliminaron

    → Dependencias: I5 (son el mismo problema)
```

### 1.5 Calidad de Vida

```
I15 pyproject.toml version 0.13.0 vs código 1.0.0 vs tags 0.29.0
    → Impacto hoy: pip, dependabot, PyPI, cualquier herramienta que lea pyproject.toml se confunde

    → Dependencias: ninguna

I16 pyproject.toml vs requirements/* desincronizados (rank-bm25, blake3 faltan; typer, coloredlogs sobran)
    → Impacto hoy: pip install -r requirements/base.txt puede fallar si falta rank-bm25
    → Dependencias: I15 (mismo archivo)

I17 773 except Exception: (degradación controlada pero sin jerarquía de excepciones)
    → Impacto hoy: no se puede distinguir entre "error esperado" y "bug real"
    → Riesgo: errores reales camuflados como degradación controlada
    → Dependencias: ninguna (ortogonal al resto)

I18 4 bare except: (error real, no degradación)
    → Impacto hoy: 4 excepciones silenciadas que pueden ocultar fallos
    → Dependencias: I17 (se resuelve con la misma política)

I19 Token GitHub en .git/config (ghp_... en texto plano)
    → Impacto hoy: cualquiera con acceso al repo local tiene el token
    → Riesgo de seguridad inmediato
    → Dependencias: ninguna (se resuelve en 5 minutos)
```

I22 5 documentos de arquitectura general superpuestos

    → Dependencias: I21 (misma limpieza)

### 2.4 Desaparecer (error de diseño confirmado)

| Componente | Por qué fue un error |
|------------|---------------------|
| `motor/intelligence/agents/` (F12) | Se diseñó una arquitectura de agentes completa (consensus, parallel, reflection, supervisor, researcher, validator, planner) antes de tener un caso de uso real. Cuando llegó el caso de uso (asistente conversacional), se rediseñó desde cero en F27. Los agentes F12 nunca se usaron en producción. Esto es ~2000 líneas de código que deberían haberse eliminado en F27. |
| `motor/intelligence/memory/` (F12) | Mismo patrón: se diseñó un sistema de memoria contextual completo antes de saber qué tipo de memoria necesitaba el producto. Llegó F26 con un diseño diferente y no se eliminó el anterior. |
| `core/mochila/providers/` | Se creó una capa de providers en `core/` cuando la intención declarada (F17, F18) era tenerlos en `motor/`. En lugar de migrar, se mantuvieron ambos. Esto duplica el esfuerzo de mantener cada provider. |
| `motor/platform/` (parcialmente) | Construir 16 archivos de infraestructura de protocolo (F28) con 0 consumidores reales fue un error de priorización. La capa de transporte debería haber empezado como 2-3 archivos y crecido con la demanda. `tracing.py` (894 líneas), `negotiator.py` (114 líneas), `serializer.py` (268 líneas) se escribieron completos antes de que cualquier módulo los necesitara. |

### Fase B — Unificar core/ → motor/ (La Fase Crítica)

**Objetivo:** `motor/` es la única capa de código activa. `core/` pasa a legacy congelado.

**IMPORTANTE:** No se trata de reescribir `core/`. Se trata de:
1. Identificar qué archivos de `core/` tienen consumidores reales en `motor/`
2. Migrar esos consumidores a usar las versiones de `motor/`
3. Marcar `core/` como congelado (no eliminar, no mantener)

| Sub-fase | Tarea | Archivos | Tiempo |
|----------|-------|----------|--------|
| B.1 | Migrar `motor/core/llm/ollama.py` de `core.config_manager.CONFIG` a `motor.core.config.UraConfig` | `motor/core/llm/ollama.py` | 1h |
| B.2 | Verificar que `motor/core/config.py` (UraConfig) cubre todas las claves que `config_manager.py` expone | `motor/core/config.py`, `core/config_manager.py` | 1h |
| B.3 | Migrar `core/memory_engine.py` a usar solo `motor/`. O congelarlo y que `motor/` no dependa de él. | `core/memory_engine.py`, `motor/cli/cmd_ura.py` | 2h |
| B.4 | Verificar que ningún archivo en `motor/` importa de `core/`. | Toda la codebase | 30 min |
| B.5 | Añadir `core/README.md` que diga: "LEGACY — Congelado. No añadir código nuevo." | `core/README.md` | 10 min |

**Resultado después de B:** 
- `motor/` no importa de `core/` (el ciclo I2 se rompe).
- `core/` puede importar de `motor/` (sentido único).
- El ciclo `motor/cli/cmd_ura.py → core.memory_engine → motor.core.*` se rompe.

**Riesgos:**
- Migrar `config_manager.py` incorrectamente podría romper la carga de configuración. Mitigación: tests de integración después de B.1.
- `core/memory_engine.py` tiene 636 líneas y 3 responsabilidades. Migrar requiere dividir, no copiar.

**Criterio de finalización:** `grep -r "from core\|import core" motor/` devuelve 0 resultados. Audit pasa.
**Tiempo total:** ~5h.

| F.5 Añadir a .pre-commit-config.yaml o ruff: prohibir bare `except:` en código nuevo | `.pre-commit-config.yaml` | 10 min |

**Riesgos:** Bajo. No se toca el código existente. Solo se establece la política para el futuro.
**Criterio de finalización:** `motor/exceptions.py` existe. 0 bare `except:` en la codebase. Ruff rule activada.
**Tiempo total:** ~3h.

### Fase G — motor/platform: Validar o Simplificar

**Objetivo:** Decidir si `motor/platform/` (16 archivos, F28) merece mantenerse o debe reducirse drásticamente.

| Tarea | Archivos | Tiempo |
|-------|----------|--------|
| G.1 Identificar consumidores reales de `motor/platform/` (más allá de `scripts/soak_test.py`) | Toda la codebase | 1h |
| G.2 Si solo hay 1-2 consumidores: reducir `motor/platform/` a lo mínimo necesario | 16 archivos | 4h |
| G.3 Simplificar `motor/platform/tracing.py` (894 líneas) separando export, sampling, sanitizer | `tracing.py` | 2h |
| G.4 Evaluar si `negotiator.py`, `serializer.py`, `delivery.py` tienen sentido sin consumidores | 3 archivos | 1h |

**Riesgos:** Si en el futuro `motor/platform/` es necesario (ej: para comunicación inter-agente), reducirlo ahora puede requerir rehacerlo después. Pero mantener 3000 LOC sin uso también es costoso.

**Decisión recomendada:** Si los únicos consumores son `scripts/soak_test.py` + tests, reducir a: `models.py` + `transport.py` + `errors.py`. El resto (tracing, negotiator, serializer, middleware, delivery, audit, registry) archivar hasta que haya un consumidor real.

**Criterio de finalización:** `motor/platform/` tiene ≤5 archivos o se ha documentado por qué deben mantenerse los 16.
**Tiempo total:** ~8h.

### Fase H — Tests + CI/CD

**Objetivo:** Que los tests existentes se ejecuten automáticamente en cada push. No se pide aumentar cobertura.

| Tarea | Archivos | Tiempo |
|-------|----------|--------|
| H.1 Activar GitHub Actions: `ci.yml` debe ejecutarse en push a main y PR | `.github/workflows/ci.yml` | 1h |
| H.2 Verificar que `ci.yml` refleja los checks reales (ruff, pytest, mypy, bandit, audit) | `.github/workflows/ci.yml` | 1h |
| H.3 Documentar dónde van los tests: `motor/tests/` para unitarios, `tests/` para integración | `AGENTS.md` o `CONTRIBUTING.md` | 30 min |
| H.4 Mover tests de `motor/` que sean realmente de integración a `tests/` (o viceversa) | ~35 archivos en `motor/tests/` | 3h |
| H.5 Añadir branch protection en GitHub: requerir CI verde para merge a main | GitHub UI (no código) | 15 min |

**Riesgos:** 
- `ci.yml` puede fallar si el entorno de GitHub Actions no tiene las mismas dependencias que el local. Mitigación: probar con un PR de prueba.
- Branch protection puede bloquear merges si CI falla constantemente. Asegurarse de que CI pase antes de activar.

**Criterio de finalización:** Push a main → GitHub Actions corre audit + tests. Si falla, no se puede mergear.
**Tiempo total:** ~6h.
