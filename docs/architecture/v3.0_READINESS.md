## Q1 — Componentes existentes que ya cubren parcialmente v3.0

| Componente v3.0 | Cobertura actual | Archivos |
|----------------|------------------|----------|
| **Goal Manager** | 🔴 No existe. Ledger guarda ejecuciones pero no hay entidad "objetivo" | — |
| **Planner** | 🟡 Parcial. `plugin_registry.run_phase()` descubre plugins, los ordena por prioridad y los ejecuta en fases. Pero no descompone objetivos en tareas | `plugin_registry.py` |
| **Task Executor** | 🟢 Alto. `PipelineEngine.run_script()`, `run_ruff()`, `run_git()` cubren la mayoría de tareas. `pipeline_refactor.py` para refactorización pesada | `engine.py`, `pipeline_refactor.py` |
| **Evaluator** | 🟡 Parcial. `PromotionPolicy` verifica ruff OK y presupuesto, pero no decide finalizar/corregir/escalar | `engine.py` (PromotionPolicy) |
| **Learning** | 🔴 No existe. No hay componente que analice el historial del Ledger | — |
| **ExecutionLedger** | 🟡 Parcial. Guarda metadatos de ejecución pero no tiene métodos para registrar decisiones, objetivos ni replanificación | `ledger.py` |

**Conclusión:** Executor está cubierto. Planner tiene base sólida. Goal Manager, Evaluator profundo y Learning son nuevo desarrollo. Ledger necesita extensión menor.

## Q3 — Acoplamientos ocultos que dificultan la autonomía como plugins

### 🔴 Crítico: plugins existentes solo aceptan args CLI

Los 19 plugins descubiertos por `plugin_registry` se ejecutan vía `subprocess.run()` con argumentos CLI. No exponen una API Python que el Task Executor pueda llamar directamente.

**Impacto:** El Task Executor debe usar `PipelineEngine.run_script()` (subprocess) en lugar de llamar a los plugins como bibliotecas.

**Solución:** No requiere cambios — el subprocess ya funciona y es el patrón establecido. La autonomía no necesita una API programática para los scripts existentes. Los plugins nuevos de autonomía pueden diseñarse con API dual (CLI + llamada directa).

### 🟡 Medio: run_phase() no retorna estructura de tareas

`run_phase()` devuelve `{"results": {"plugin_name": {"status": "ok", ...}}}` pero no una lista de "tareas completadas" que el Evaluator pueda consumir directamente.

**Solución:** El Planner puede envolver `run_phase()` y transformar la salida al formato de tareas que necesite el Evaluator.

### 🟡 Medio: plugin_registry no discrimina por dominio

Un plugin de auditoría aparece tanto en la tuneladora de mejora como en el sistema de autonomía. No hay campo `capability` que permita filtrar.

**Solución:** Añadir `"capability": "autonomy"` al PLUGIN. Los plugins sin este campo se consideran de infraestructura (comportamiento actual).

### 🟢 Bajo: ExecutionLedger crece sin retención

Cada ejecución añade un JSON a `.nervioso/ledger/`. Sin rotación, el Learning Plugin tendrá que procesar cientos de archivos.

**Solución:** No bloqueante. Se abordará cuando el volumen lo justifique.

## Q5 — Deuda técnica que bloqueará v3.0 si no se resuelve antes

### 🔴 Bloqueante (resolver antes de empezar)

| Deuda | Impacto | Solución | Esfuerzo |
|-------|---------|----------|----------|
| Ledger no registra decisiones | Learning no tiene datos para analizar | Añadir `set_goal()`, `set_decision()`, `set_replan_reason()` | ~30 min |
| PromotionPolicy no evalúa objetivos | Evaluator no puede decidir si un objetivo se cumplió | Añadir `evaluate_goal()` | ~15 min |
| plugin_registry no filtra por capacidad | Plugins de autonomía aparecen en mantenimiento | Añadir campo `capability` | ~10 min |

### 🟡 No bloqueante (puede esperar)

| Deuda | Impacto |
|-------|---------|
| Plugins solo aceptan args CLI | Task Executor usa subprocess (solución actual, funciona) |
| Ledger sin rotación | Crecimiento de archivos, abordar cuando sea problema |
| Compactadora/scanner tienen bugs pre-existentes | El Executor ya los maneja (warn + continue) |
