# Análisis de Readiness — v3.0 Autonomía

**Fecha:** 2026-07-20  
**Base:** Infraestructura v2.3 congelada (ADR-030)  
**Método:** Análisis con evidencia del código actual

---

## Q1 — Componentes existentes vs responsabilidades v3.0

### Goal Manager (crear, priorizar, suspender, cancelar objetivos)

**Cobertura: 0%.** Ningún componente implementa estas funciones. No existe persistencia de "objetivos" como entidad separada. El `ExecutionLedger` guarda ejecuciones completas, no objetivos individuales con estado y prioridad.

**Evidencia:** No hay `create_goal()`, `prioritize()`, `suspend()` ni `cancel()` en ningún archivo de `scripts/pro/tuneladora/`.

### Planner (descomponer objetivos, ordenar tareas, estimar)

**Cobertura: 40%.** `plugin_registry.py` ya implementa parte del planificador:

- `plugin_registry.py:48` → `_resolve_dependencies()` ordena plugins por prioridad y verifica dependencias
- `plugin_registry.py:66` → `sorted_plugins = sorted(plugins.items(), key=lambda x: x[1].get("priority", 500))`
- `plugin_registry.py:98` → `run_phase()` ejecuta plugins de una fase respetando ese orden

**Lo que falta:** `run_phase()` recibe un string de fase (`"pre"`, `"refactor"`, `"post"`) no un objetivo. No hay descomposición de objetivos → tareas.

### Task Executor (ejecutar tareas, reintentar, seleccionar herramientas)

**Cobertura: 80%.** El PipelineEngine v2.3 ya proporciona:

| API | Archivo:Línea | Tareas que cubre |
|-----|---------------|------------------|
| `run_script()` | `engine.py:118` | Ejecutar cualquier script del proyecto |
| `run_ruff()` | `engine.py:142` | Calidad de código |
| `run_git()` | `engine.py:163` | Git diff, commit, log |
| `health_ollama()` | `engine.py:174` | Verificar conectividad |
| `health_disk()` | `engine.py:186` | Verificar recursos |
| `WorkerManager.run_workers()` | `worker_manager.py:53` | Workers paralelos |

**Lo que falta:** No hay `run_task()`, `select_tool()`, `retry_policy()` como métodos específicos. El Executor de v3.0 envolverá estas APIs existentes.

### Evaluator (medir cumplimiento, decidir finalizar/corregir/escalar)

**Cobertura: 30%.** `PromotionPolicy` (`engine.py:60`) proporciona:

- `record(check, ok, detail)` → `engine.py:78`
- `can_promote` → `engine.py:82`
- `check_budget(files, lines)` → `engine.py:72`

**Lo que falta:** `can_promote` evalúa checks planos (ruff OK, budget OK). No existe `evaluate_goal(goal, results)` que decida finalizar/corregir/escalar. No existe `decide()`.

### Learning Plugin (analizar histórico, patrones, propuestas)

**Cobertura: 0%.** No existe ningún componente que lea el histórico del ExecutionLedger y extraiga patrones de éxito/fallo. Es desarrollo 100% nuevo.

## Q6 — Dependencias circulares o acoplamientos

### No hay dependencias circulares

- `motor/cli/cmd_ura.py` importa scripts/pro/ (unidireccional, ya actualizado tras migración v2)
- `scripts/pro/tuneladora/` no importa nada de motor/ ni core/
- `plugin_registry.py` no importa tuneladora/

### Acoplamiento conocido: plugins CLI-only

Los 16 plugins reutilizables solo aceptan argumentos CLI. No exponen API Python. El Task Executor debe usar `PipelineEngine.run_script()` (subprocess) para ejecutarlos. **Esto no es un problema** — el subprocess funciona y es el patrón establecido desde v2.0.

### Riesgo real: mezcla de plugins de infraestructura y autonomía

`plugin_registry.run_phase("post")` descubre TANTO plugins de mantenimiento (f821_watch, inspectores) COMO plugins que podrían ser de autonomía. Sin un campo `capability`, no hay forma de discriminar.

**Solución:** Añadir `"capability": "autonomy"` al PLUGIN. Los plugins sin este campo se consideran de infraestructura (comportamiento actual).
